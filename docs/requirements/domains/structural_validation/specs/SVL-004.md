## Summary

`L2BlockHeader::validate()` must reject blocks with timestamps too far in the future to prevent time manipulation attacks. This is adopted from Chia `block_header_validation.py` Check 26a.

## Specification

```rust
const MAX_FUTURE_TIMESTAMP_SECONDS: u64 = 300; // 5 minutes

// Step 5: Timestamp future bound
let now = SystemTime::now()
    .duration_since(UNIX_EPOCH)
    .unwrap()
    .as_secs();

if self.timestamp > now + MAX_FUTURE_TIMESTAMP_SECONDS {
    return Err(BlockError::TimestampTooFarInFuture {
        timestamp: self.timestamp,
        max_allowed: now + MAX_FUTURE_TIMESTAMP_SECONDS,
    });
}
```

- The default value of `MAX_FUTURE_TIMESTAMP_SECONDS` is 300 seconds (5 minutes).
- The comparison is strict greater-than: a timestamp exactly at `now + 300` is accepted.
- The current time (`now`) is sampled at validation time.

## Acceptance Criteria

- [x] `validate()` returns `Ok` when `timestamp == now` (integration tests use `validate_with_dfsp_activation_at_unix` with fixed `now_secs`; same inequality as `validate()`).
- [x] `validate()` returns `Ok` when `timestamp == now + MAX_FUTURE_TIMESTAMP_SECONDS` (boundary).
- [x] `validate()` returns `Err(BlockError::TimestampTooFarInFuture)` when `timestamp == now + MAX_FUTURE_TIMESTAMP_SECONDS + 1`.
- [x] `validate()` returns `Ok` when timestamp is in the past.
- [x] The default `MAX_FUTURE_TIMESTAMP_SECONDS` value is 300.

## Implementation Notes

- This check is adopted from Chia `block_header_validation.py` Check 26a and serves the same purpose: preventing a proposer from claiming a future timestamp to gain ordering advantages.
- Test implementations should use a controllable time source or inject the "now" value to avoid flaky tests.
- No minimum timestamp check is performed at this layer; past timestamps are accepted.
