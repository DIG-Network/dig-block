# EXE-004: Condition Parsing and Assertion Checking

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#exe-004-condition-parsing-and-assertion-checking)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.4.4

## Summary

Conditions from CLVM output MUST be validated in two passes. Pass 1 collects coin/announcement creations and reserve fees. Pass 2 validates assertions against the collected state. Height/time assertions are collected but deferred to Tier 3.

## Specification

```rust
use chia_sdk_types::Condition;

struct ConditionState {
    created_coins: Vec<Coin>,
    coin_announcements: HashSet<Bytes32>,
    puzzle_announcements: HashSet<Bytes32>,
    reserve_fee: u64,
    spending_coin_ids: HashSet<Bytes32>,
    spending_puzzle_hashes: HashSet<Bytes32>,
    pending_assertions: Vec<PendingAssertion>,
}

fn process_conditions(
    conditions: &[Condition],
    coin_spend: &CoinSpend,
) -> Result<ConditionState, BlockError> {
    let mut state = ConditionState::default();

    // Pass 1: Collect outputs
    for condition in conditions {
        match condition {
            Condition::CreateCoin(cc) => {
                state.created_coins.push(Coin::new(
                    coin_spend.coin.coin_id(),
                    cc.puzzle_hash,
                    cc.amount,
                ));
            }
            Condition::CreateCoinAnnouncement(ann) => {
                let id = hash_coin_announcement(coin_spend.coin.coin_id(), &ann.message);
                state.coin_announcements.insert(id);
            }
            Condition::CreatePuzzleAnnouncement(ann) => {
                let id = hash_puzzle_announcement(coin_spend.coin.puzzle_hash, &ann.message);
                state.puzzle_announcements.insert(id);
            }
            Condition::ReserveFee(rf) => {
                state.reserve_fee += rf.amount;
            }
            // Height/time assertions deferred to Tier 3
            Condition::AssertHeightAbsolute(_)
            | Condition::AssertHeightRelative(_)
            | Condition::AssertSecondsAbsolute(_)
            | Condition::AssertSecondsRelative(_)
            | Condition::AssertBeforeHeightAbsolute(_)
            | Condition::AssertBeforeHeightRelative(_)
            | Condition::AssertBeforeSecondsAbsolute(_)
            | Condition::AssertBeforeSecondsRelative(_) => {
                state.pending_assertions.push(PendingAssertion::from(condition, coin_spend));
            }
            // ASSERT_EPHEMERAL: coin must be created in same block (checked in Tier 3)
            Condition::AssertEphemeral => {
                state.pending_assertions.push(PendingAssertion::ephemeral(coin_spend));
            }
            _ => {}
        }
    }

    // Pass 2: Validate assertions
    for condition in conditions {
        match condition {
            Condition::AssertCoinAnnouncement(a) => {
                if !state.coin_announcements.contains(&a.announcement_id) {
                    return Err(BlockError::AssertionFailed("coin announcement".into()));
                }
            }
            Condition::AssertPuzzleAnnouncement(a) => {
                if !state.puzzle_announcements.contains(&a.announcement_id) {
                    return Err(BlockError::AssertionFailed("puzzle announcement".into()));
                }
            }
            Condition::AssertConcurrentSpend(a) => {
                if !state.spending_coin_ids.contains(&a.coin_id) {
                    return Err(BlockError::AssertionFailed("concurrent spend".into()));
                }
            }
            Condition::AssertConcurrentPuzzle(a) => {
                if !state.spending_puzzle_hashes.contains(&a.puzzle_hash) {
                    return Err(BlockError::AssertionFailed("concurrent puzzle".into()));
                }
            }
            Condition::AssertMyCoinId(a) => {
                if a.coin_id != coin_spend.coin.coin_id() {
                    return Err(BlockError::AssertionFailed("my coin id".into()));
                }
            }
            Condition::AssertMyParentId(a) => {
                if a.parent_id != coin_spend.coin.parent_coin_info {
                    return Err(BlockError::AssertionFailed("my parent id".into()));
                }
            }
            Condition::AssertMyPuzzlehash(a) => {
                if a.puzzle_hash != coin_spend.coin.puzzle_hash {
                    return Err(BlockError::AssertionFailed("my puzzlehash".into()));
                }
            }
            Condition::AssertMyAmount(a) => {
                if a.amount != coin_spend.coin.amount {
                    return Err(BlockError::AssertionFailed("my amount".into()));
                }
            }
            _ => {}
        }
    }

    Ok(state)
}
```

## Acceptance Criteria

- [ ] Pass 1 collects CREATE_COIN, CREATE_COIN_ANNOUNCEMENT, CREATE_PUZZLE_ANNOUNCEMENT, RESERVE_FEE
- [ ] Pass 2 validates ASSERT_COIN_ANNOUNCEMENT against collected announcements
- [ ] Pass 2 validates ASSERT_PUZZLE_ANNOUNCEMENT against collected announcements
- [ ] Pass 2 validates ASSERT_CONCURRENT_SPEND and ASSERT_CONCURRENT_PUZZLE
- [ ] Pass 2 validates ASSERT_MY_COIN_ID, ASSERT_MY_PARENT_ID, ASSERT_MY_PUZZLEHASH, ASSERT_MY_AMOUNT
- [ ] Height/time assertions are collected into pending_assertions (deferred to Tier 3)
- [ ] ASSERT_EPHEMERAL is collected into pending_assertions (deferred to Tier 3)
- [ ] Uses `chia-sdk-types::Condition` directly

## Implementation Notes

- The two-pass approach is necessary because assertions in Pass 2 reference state created in Pass 1.
- ASSERT_CONCURRENT_SPEND and ASSERT_CONCURRENT_PUZZLE require visibility across all spends in the block, so the spending sets must be populated at the block level before Pass 2.
- This logic may be partially inside dig-clvm. The spec files document the logical requirement regardless of which crate implements it.
