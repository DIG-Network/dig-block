# EXE-001: validate_execution API

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#exe-001-validate_execution-api)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.4

## Summary

L2Block MUST provide a `validate_execution` method that accepts a `ValidationConfig` (from dig-clvm) and a `genesis_challenge`, returning either a fully populated `ExecutionResult` or a `BlockError`. Internally it creates a `clvmr::Allocator` and processes each `SpendBundle` in block order.

## Specification

```rust
impl L2Block {
    pub fn validate_execution(
        &self,
        clvm_config: &ValidationConfig,
        genesis_challenge: &Bytes32,
    ) -> Result<ExecutionResult, BlockError> {
        let mut allocator = clvmr::Allocator::new();
        let mut all_additions = Vec::new();
        let mut all_removals = Vec::new();
        let mut all_pending = Vec::new();
        let mut total_cost: Cost = 0;
        let mut total_fees: u64 = 0;
        let mut receipts = Vec::new();

        for spend_bundle in &self.body.spend_bundles {
            let result = dig_clvm::validate_spend_bundle(
                &mut allocator,
                clvm_config,
                genesis_challenge,
                spend_bundle,
            )
            .map_err(|e| map_validation_error(e))?;

            all_additions.extend(result.additions);
            all_removals.extend(result.removals);
            all_pending.extend(result.pending_assertions);
            total_cost += result.conditions.cost;
            total_fees += result.fees;
            receipts.push(result.receipt);
        }

        // EXE-006: Fee consistency
        if total_fees != self.header.total_fees {
            return Err(BlockError::FeesMismatch);
        }

        // EXE-007: Cost consistency
        if total_cost != self.header.total_cost {
            return Err(BlockError::CostMismatch);
        }

        Ok(ExecutionResult {
            additions: all_additions,
            removals: all_removals,
            pending_assertions: all_pending,
            total_cost,
            total_fees,
            receipts,
        })
    }
}
```

## Acceptance Criteria

- [ ] `validate_execution` method exists on `L2Block` with the specified signature
- [ ] `ValidationConfig` is sourced from `dig-clvm`, not defined locally
- [ ] A `clvmr::Allocator` is created internally per invocation
- [ ] Each `SpendBundle` is processed in block order
- [ ] Returns `Ok(ExecutionResult)` on success
- [ ] Returns `Err(BlockError)` on any validation failure

## Implementation Notes

- The `ValidationConfig` type encapsulates CLVM cost limits, hard fork heights, and other consensus parameters managed by dig-clvm.
- Block order processing is critical because ephemeral coins created in one spend may be consumed in a subsequent spend within the same block.
