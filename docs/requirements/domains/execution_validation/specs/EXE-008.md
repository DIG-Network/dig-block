# EXE-008: ExecutionResult Output Type

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#exe-008-executionresult-output-type)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.4.7

## Summary

`ExecutionResult` MUST contain all validated outputs from Tier 2 execution, carrying them forward to Tier 3 (state validation). The struct serves as the bridge between execution validation and state validation.

## Specification

```rust
/// Validated output from Tier 2 execution validation.
/// Passed to Tier 3 (state validation) for coin state checks
/// and height/time lock evaluation.
pub struct ExecutionResult {
    /// New coins created by CREATE_COIN conditions
    pub additions: Vec<Coin>,

    /// Coin IDs of coins consumed (spent) in this block
    pub removals: Vec<Bytes32>,

    /// Height/time assertions deferred from Tier 2 to Tier 3
    pub pending_assertions: Vec<PendingAssertion>,

    /// Total CLVM execution cost across all spend bundles
    pub total_cost: Cost,

    /// Total fees across all spend bundles (input - output)
    pub total_fees: u64,

    /// Per-spend-bundle receipts for event logging
    pub receipts: Vec<Receipt>,
}
```

## Acceptance Criteria

- [ ] `ExecutionResult` struct exists with `additions: Vec<Coin>`
- [ ] `ExecutionResult` struct contains `removals: Vec<Bytes32>`
- [ ] `ExecutionResult` struct contains `pending_assertions: Vec<PendingAssertion>`
- [ ] `ExecutionResult` struct contains `total_cost: Cost`
- [ ] `ExecutionResult` struct contains `total_fees: u64`
- [ ] `ExecutionResult` struct contains `receipts: Vec<Receipt>`
- [ ] Type is used as the return value from `validate_execution`
- [ ] Type is accepted as input by `validate_state`

## Implementation Notes

- `PendingAssertion` encapsulates height/time assertions that require chain context (current height, timestamps, coin confirmed heights) which is only available during Tier 3 validation.
- `Receipt` captures per-spend-bundle metadata for logging, indexing, and event notification purposes.
- `Coin` and `Bytes32` are from `chia-protocol`. `Cost` is typically a `u64` alias.
