# EXE-003: CLVM Execution via dig-clvm

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#exe-003-clvm-execution-via-dig-clvm)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.4.3

## Summary

Each SpendBundle MUST be validated by calling `dig_clvm::validate_spend_bundle()`. This function handles CLVM execution via `chia-consensus::run_spendbundle()`, condition parsing, and cost limit enforcement. dig-block MUST NOT call chia-consensus directly.

## Specification

```rust
// dig-block delegates all CLVM execution to dig-clvm
use dig_clvm::{validate_spend_bundle, ValidationConfig, ValidationError};

fn execute_spend_bundle(
    allocator: &mut Allocator,
    config: &ValidationConfig,
    genesis_challenge: &Bytes32,
    spend_bundle: &SpendBundle,
) -> Result<SpendResult, BlockError> {
    dig_clvm::validate_spend_bundle(allocator, config, genesis_challenge, spend_bundle)
        .map_err(|e| map_validation_error(e))
}

fn map_validation_error(err: ValidationError) -> BlockError {
    match err {
        ValidationError::PuzzleHashMismatch { .. } => BlockError::PuzzleHashMismatch,
        ValidationError::SignatureFailed => BlockError::SignatureFailed,
        ValidationError::CostExceeded => BlockError::CostExceeded,
        ValidationError::ConservationFailed => BlockError::ConservationFailed,
        ValidationError::ReserveFeeFailed => BlockError::ReserveFeeFailed,
        ValidationError::InvalidClvm(msg) => BlockError::InvalidClvm(msg),
        // Map other variants as appropriate
        other => BlockError::ExecutionError(other.to_string()),
    }
}
```

## Acceptance Criteria

- [ ] `dig_clvm::validate_spend_bundle()` is called for each `SpendBundle`
- [ ] dig-block does NOT import or call `chia-consensus` directly
- [ ] All `dig-clvm::ValidationError` variants are mapped to `BlockError` variants
- [ ] CLVM execution, condition parsing, and cost enforcement are fully delegated
- [ ] No CLVM-level logic is duplicated in dig-block

## Implementation Notes

- The architectural boundary between dig-block and dig-clvm is critical. dig-block owns block-level validation (fee/cost totals, state transitions). dig-clvm owns spend-level validation (CLVM execution, signature verification, per-spend conservation).
- A `grep` or CI lint should enforce that `chia_consensus` does not appear in dig-block's imports.
