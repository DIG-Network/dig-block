## Summary

Defines the `CheckpointStatus` enum representing the lifecycle of a checkpoint epoch, from initial pending state through collection, winner selection, finalization, or failure.

## Specification

### Enum Variants

```rust
pub enum CheckpointStatus {
    Pending,
    Collecting,
    WinnerSelected {
        winner_hash: Bytes32,
        winner_score: u64,
    },
    Finalized {
        winner_hash: Bytes32,
        l1_height: u32,
    },
    Failed,
}
```

### Variant Descriptions

| Variant          | Description                                                            |
|------------------|------------------------------------------------------------------------|
| `Pending`        | Epoch has not yet started checkpoint collection.                       |
| `Collecting`     | Validators are submitting checkpoint proposals for the epoch.          |
| `WinnerSelected` | A winning checkpoint has been selected but not yet confirmed on L1.    |
| `Finalized`      | The winning checkpoint has been confirmed on L1 at `l1_height`.        |
| `Failed`         | The checkpoint process failed (e.g., insufficient participation).      |

## Acceptance Criteria

- [ ] `CheckpointStatus` defines exactly five variants.
- [ ] `WinnerSelected` carries `winner_hash: Bytes32` and `winner_score: u64`.
- [ ] `Finalized` carries `winner_hash: Bytes32` and `l1_height: u32`.
- [ ] `Pending`, `Collecting`, and `Failed` carry no data.
- [ ] Pattern matching works correctly on all variants.

## Implementation Notes

- `WinnerSelected` and `Finalized` are data-carrying enum variants, used to track the winning checkpoint through the finalization process.
- The transition order is typically: `Pending` -> `Collecting` -> `WinnerSelected` -> `Finalized`, with `Failed` as an alternative terminal state.
