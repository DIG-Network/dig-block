# STV-002: Coin Existence Checks

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#stv-002-coin-existence-checks)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.5.1

## Summary

For each removal, the coin MUST exist in the coin state database or be an ephemeral coin created in the same block. Already-spent coins MUST be rejected.

## Specification

```rust
fn check_coin_existence(
    &self,
    exec: &ExecutionResult,
    coins: &dyn CoinLookup,
) -> Result<(), BlockError> {
    let ephemeral_ids: HashSet<Bytes32> = exec.additions
        .iter()
        .map(|c| c.coin_id())
        .collect();

    for removal_id in &exec.removals {
        match coins.get_coin_state(removal_id) {
            Some(coin_state) => {
                // Coin exists - check it hasn't already been spent
                if coin_state.spent_height.is_some() {
                    return Err(BlockError::CoinAlreadySpent {
                        coin_id: *removal_id,
                    });
                }
            }
            None => {
                // Coin not in database - check if ephemeral
                if !ephemeral_ids.contains(removal_id) {
                    return Err(BlockError::CoinNotFound {
                        coin_id: *removal_id,
                    });
                }
            }
        }
    }

    Ok(())
}
```

## Acceptance Criteria

- [ ] Each removal is looked up via `coins.get_coin_state(coin_id)`
- [ ] If `Some` and `spent_height.is_some()`, reject with `CoinAlreadySpent`
- [ ] If `None`, check if coin is ephemeral (present in `exec.additions`)
- [ ] If not ephemeral and not in database, reject with `CoinNotFound`
- [ ] Ephemeral coins (created and spent in same block) pass existence check
- [ ] Chia parity with Check 15

## Implementation Notes

- Ephemeral coins are coins that are both created (via CREATE_COIN) and spent within the same block. They never exist in the persistent coin state database.
- The order of spend bundles matters: a coin created by an earlier bundle can be spent by a later bundle in the same block.
- The `ASSERT_EPHEMERAL` condition (collected in EXE-004, deferred to Tier 3) requires that the coin producing the condition was created in the same block. This is verified by checking that the coin's ID appears in `exec.additions`. If it does not, reject with `BlockError::AssertionFailed`.
