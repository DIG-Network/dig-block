# STV-006: Proposer Signature Verification

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#stv-006-proposer-signature-verification)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.5.5

## Summary

The proposer's BLS signature over the block header hash MUST be verified against the provided proposer public key. This ensures the block was produced by an authorized proposer.

## Specification

```rust
use chia_bls::{verify, PublicKey, Signature};

fn verify_proposer_signature(
    &self,
    proposer_pubkey: &PublicKey,
) -> Result<(), BlockError> {
    let header_hash = self.header.hash();
    let signature = &self.proposer_signature;

    if !verify(proposer_pubkey, &header_hash, signature) {
        return Err(BlockError::InvalidProposerSignature);
    }

    Ok(())
}
```

## Acceptance Criteria

- [ ] `verify(proposer_pubkey, header.hash(), block.proposer_signature)` is called
- [ ] Uses `chia-bls::verify` for BLS signature verification
- [ ] Valid signature passes
- [ ] Invalid signature rejects with `BlockError::InvalidProposerSignature`
- [ ] Wrong public key rejects with `BlockError::InvalidProposerSignature`

## Implementation Notes

- The proposer public key is provided externally (from the epoch's proposer set or validator registry). dig-block does not determine who the proposer should be; it only verifies the signature.
- `header.hash()` produces the canonical hash of the block header, which is the message signed by the proposer.
