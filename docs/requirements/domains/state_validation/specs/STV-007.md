# STV-007: State Root Verification

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#stv-007-state-root-verification)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.5.6

## Summary

After applying all additions and removals to the parent state, the recomputed Merkle root MUST match `header.state_root`. This is the final verification step, confirming that the block's declared state transition is correct.

## Specification

```rust
fn compute_and_verify_state_root(
    &self,
    exec: &ExecutionResult,
    coins: &dyn CoinLookup,
) -> Result<Bytes32, BlockError> {
    // Start from parent state root
    let mut state_tree = coins.get_state_tree();

    // Mark removals as spent at this block's height
    for removal_id in &exec.removals {
        state_tree.mark_spent(removal_id, self.header.height)?;
    }

    // Insert additions into the state tree
    for addition in &exec.additions {
        state_tree.insert(CoinState {
            coin: addition.clone(),
            created_height: Some(self.header.height),
            spent_height: None,
            timestamp: self.header.timestamp,
        })?;
    }

    // Recompute the Merkle root
    let computed_state_root = state_tree.root();

    if computed_state_root != self.header.state_root {
        return Err(BlockError::InvalidStateRoot {
            computed: computed_state_root,
            header: self.header.state_root,
        });
    }

    Ok(computed_state_root)
}
```

## Acceptance Criteria

- [ ] Starts from parent state (accessible via CoinLookup)
- [ ] All removals are marked as spent in the state tree
- [ ] All additions are inserted into the state tree
- [ ] Merkle root is recomputed after all mutations
- [ ] `computed_state_root` MUST equal `header.state_root`
- [ ] Mismatch rejects with `BlockError::InvalidStateRoot`
- [ ] Returns `computed_state_root` on success

## Implementation Notes

- The state tree is a Merkle tree (likely a sparse Merkle tree or Patricia trie) that allows efficient root computation after incremental updates.
- The order of operations (removals then additions, or vice versa) must be deterministic and match the proposer's computation to produce the same root.
- Ephemeral coins that are both added and spent in the same block must be handled correctly: they are inserted and then immediately marked as spent.
