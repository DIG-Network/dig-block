# STV-004: Addition Non-Existence

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#stv-004-addition-non-existence)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 7.5.3

## Summary

For each addition (new coin created by CREATE_COIN), the coin MUST NOT already exist in the coin state database, unless it is ephemeral (created and spent within the same block).

## Specification

```rust
fn check_addition_uniqueness(
    &self,
    exec: &ExecutionResult,
    coins: &dyn CoinLookup,
) -> Result<(), BlockError> {
    let removal_ids: HashSet<Bytes32> = exec.removals
        .iter()
        .cloned()
        .collect();

    for addition in &exec.additions {
        let coin_id = addition.coin_id();

        if let Some(_existing) = coins.get_coin_state(&coin_id) {
            // Exception: ephemeral coins created and spent in the same block
            // These appear in both additions and removals
            if !removal_ids.contains(&coin_id) {
                return Err(BlockError::CoinAlreadyExists {
                    coin_id,
                });
            }
        }
    }

    Ok(())
}
```

## Acceptance Criteria

- [ ] Each addition is checked via `coins.get_coin_state(addition.coin_id())`
- [ ] If coin already exists in database, reject with `BlockError::CoinAlreadyExists`
- [ ] Exception: ephemeral coins (created and spent in same block) are allowed
- [ ] Non-ephemeral duplicate coin IDs are rejected

## Implementation Notes

- Coin IDs are deterministic: `SHA256(parent_coin_id || puzzle_hash || amount)`. A collision would mean the same parent created two coins with identical puzzle_hash and amount, which is prevented by Chia consensus rules (duplicate CREATE_COIN outputs are forbidden).
- The ephemeral exception handles the case where a coin is created and immediately consumed within the same block.
