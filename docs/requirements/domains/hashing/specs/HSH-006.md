# HSH-006: Filter Hash Construction

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 3.6](../../../../resources/SPEC.md)

## Summary

The filter_hash enables light clients to efficiently test whether a block is relevant to their watched addresses without downloading the full block. It is computed as SHA-256 of a BIP158 compact filter built from the puzzle_hashes of all additions and the coin_ids of all removals, using Golomb-Rice coding.

## Specification

filter_hash MUST be computed as follows:

```rust
pub fn compute_filter_hash(
    additions: &[Coin],
    removals: &[Bytes32],
) -> Bytes32 {
    let mut filter_elements: Vec<&[u8]> = Vec::new();

    // Include puzzle_hash of each addition
    for coin in additions {
        filter_elements.push(coin.puzzle_hash.as_ref());
    }

    // Include coin_id of each removal
    for coin_id in removals {
        filter_elements.push(coin_id.as_ref());
    }

    // Build BIP158 compact filter with Golomb-Rice coding
    let filter = build_gcs_filter(&filter_elements);

    // filter_hash = SHA-256(filter_bytes)
    sha256(&filter)
}
```

Key rules:
- Filter input MUST include the puzzle_hash of each addition
- Filter input MUST include the coin_id of each removal
- The filter MUST be Golomb-Rice coded (BIP158 compact block filter)
- filter_hash MUST be SHA-256 of the encoded filter bytes
- Enables light client membership testing (can test if a puzzle_hash or coin_id is in the filter)

## Acceptance Criteria

- [ ] Filter includes puzzle_hash of every addition
- [ ] Filter includes coin_id of every removal
- [ ] Filter uses Golomb-Rice coding per BIP158
- [ ] filter_hash is SHA-256 of the encoded filter
- [ ] Light client can test membership of a puzzle_hash against the filter
- [ ] Light client can test membership of a coin_id against the filter
- [ ] Empty block (no additions, no removals) produces a deterministic filter_hash

## Implementation Notes

- BIP158 defines Golomb-Coded Sets (GCS) for compact block filters
- The filter allows false positives but not false negatives -- if a puzzle_hash is in the block, the filter will always report it as present
- Light clients download only the filter_hash and the filter itself, not the full block
- If the filter matches a watched address, the light client fetches the full block for detailed processing
- The Golomb-Rice parameter (P) should match BIP158 defaults
