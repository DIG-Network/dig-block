## Summary

Any `L2Block` produced by `BlockBuilder.build()` must pass `L2Block::validate_structure()` without error. This is the "build correct, validate everything" design principle: the builder computes all derived fields correctly by construction, and the validator independently verifies them.

## Specification

This requirement is a correctness invariant rather than a specific code path. It asserts:

```rust
// For any valid builder configuration:
let block = builder.build(state_root, receipts_root, &signer)?;
assert!(block.validate_structure().is_ok());
```

The guarantee holds because `build()` computes every derived field using the same algorithms that `validate_structure()` uses to verify them:

| Field                 | Built by                          | Verified by (SVL-005/006)         |
|-----------------------|-----------------------------------|-----------------------------------|
| spend_bundle_count    | `spend_bundles.len()`             | Count agreement check             |
| additions_count       | Sum of additions across bundles   | Count agreement check             |
| removals_count        | Sum of coin spends across bundles | Count agreement check             |
| slash_proposal_count  | `slash_proposal_payloads.len()`   | Count agreement check             |
| spends_root           | `MerkleTree::new().root()`        | Merkle root recomputation         |
| additions_root        | `compute_merkle_set_root()`       | Merkle root recomputation         |
| removals_root         | `compute_merkle_set_root()`       | Merkle root recomputation         |
| filter_hash           | `build_gcs_filter().hash()`       | BIP158 filter recomputation       |
| slash_proposals_root  | `compute_slash_proposals_root()`  | Merkle root recomputation         |
| block_size            | Two-pass serialization            | Serialized size check             |
| version               | Height-based auto-detection       | Version check (SVL-001)           |

If `build()` and `validate_structure()` ever disagree, it indicates a bug in one or both implementations.

## Acceptance Criteria

- [ ] A block built with zero spend bundles and zero slash proposals passes `validate_structure()`.
- [ ] A block built with one spend bundle passes `validate_structure()`.
- [ ] A block built with multiple spend bundles passes `validate_structure()`.
- [ ] A block built with slash proposals passes `validate_structure()`.
- [ ] A block built with all optional fields set passes `validate_structure()`.
- [ ] A V1 block (pre-activation) built by the builder passes `validate_structure()`.
- [ ] A V2 block (post-activation) built by the builder passes `validate_structure()`.

## Implementation Notes

- This requirement is primarily verified through round-trip integration tests.
- Any failure of this invariant should be treated as a critical bug, as it means the network could reject blocks produced by its own builder.
- Property-based testing (e.g., with `proptest` or `quickcheck`) is well-suited for this requirement, as it can generate diverse builder configurations and verify the invariant holds for all of them.
