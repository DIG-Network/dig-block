## Summary

Any `L2Block` **successfully returned** from `BlockBuilder::build()` / `build_with_dfsp_activation()` must pass `L2Block::validate_structure()` without error. (`build()` rejects zero spend bundles with [`BuilderError::EmptyBlock`](../../error_types/specs/ERR-004.md) — there is no block value to validate.) This is the "build correct, validate everything" design principle: the builder computes all derived fields correctly by construction, and the validator independently verifies them.

## Specification

This requirement is a correctness invariant rather than a specific code path. It asserts:

```rust
// For any builder configuration that yields Ok (≥1 spend bundle):
let block = builder.build(state_root, receipts_root, &signer)?;
assert!(block.validate_structure().is_ok());
```

The guarantee holds because `build()` computes every derived field using the same algorithms that `validate_structure()` uses to verify them:

| Field                 | Built by                          | Verified by (SVL-005/006)         |
|-----------------------|-----------------------------------|-----------------------------------|
| spend_bundle_count    | `spend_bundles.len()`             | Count agreement check             |
| additions_count       | Sum of additions across bundles   | Count agreement check             |
| removals_count        | Sum of coin spends across bundles | Count agreement check             |
| slash_proposal_count  | `slash_proposal_payloads.len()`   | Count agreement check             |
| spends_root           | `MerkleTree::new().root()`        | Merkle root recomputation         |
| additions_root        | `compute_merkle_set_root()`       | Merkle root recomputation         |
| removals_root         | `compute_merkle_set_root()`       | Merkle root recomputation         |
| filter_hash           | `build_gcs_filter().hash()`       | BIP158 filter recomputation       |
| slash_proposals_root  | `compute_slash_proposals_root()`  | Merkle root recomputation         |
| block_size            | Two-pass serialization            | Serialized size check             |
| version               | Height-based auto-detection       | (SVL-001 applies to `L2BlockHeader::validate`, not `validate_structure`) |

If `build()` and `validate_structure()` ever disagree on an `Ok` block, it indicates a bug in one or both implementations.

## Acceptance Criteria

- [x] `build()` with zero spend bundles does not return a block (`EmptyBlock`); an empty **body** with a header synced to that body passes `validate_structure()` (SVL-005/006 reference case).
- [x] A block built with one spend bundle passes `validate_structure()`.
- [x] A block built with multiple spend bundles passes `validate_structure()`.
- [x] A block built with slash proposals passes `validate_structure()`.
- [x] A block built with all optional fields set passes `validate_structure()`.
- [x] A V1 block (default DFSP activation sentinel) built by the builder passes `validate_structure()`.
- [x] A V2 block (`build_with_dfsp_activation` post-activation with DFSP roots) built by the builder passes `validate_structure()`.

## Implementation Notes

- This requirement is primarily verified through round-trip integration tests.
- Any failure of this invariant should be treated as a critical bug, as it means the network could reject blocks produced by its own builder.
- Property-based testing (e.g., with `proptest` or `quickcheck`) can extend coverage later; the dedicated integration crate currently uses explicit matrix tests (no extra dev-deps).
