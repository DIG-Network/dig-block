# SER-004: Serde Default Attributes

- **Normative:** [NORMATIVE.md](../NORMATIVE.md#ser-004-serde-default-attributes)
- **Verification:** [VERIFICATION.md](../VERIFICATION.md)
- **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
- **Spec:** SPEC Section 8.2

## Summary

Fields added in later protocol versions MUST use `#[serde(default)]` or `#[serde(default = "...")]` to ensure backwards compatibility when deserializing data serialized by older protocol versions.

## Specification

```rust
#[derive(Serialize, Deserialize)]
pub struct L2BlockHeader {
    pub version: u32,
    pub height: u64,
    pub epoch: u64,
    pub parent_hash: Bytes32,
    pub state_root: Bytes32,
    pub transactions_root: Bytes32,
    pub receipts_root: Bytes32,
    pub timestamp: u64,
    pub total_cost: Cost,
    pub total_fees: u64,
    pub transaction_count: u32,

    // Fields added in protocol version 2+
    #[serde(default)]
    pub filter_hash: Bytes32,

    #[serde(default)]
    pub extension_data: Bytes32,

    #[serde(default)]
    pub l1_inclusion_proof: Option<L1Proof>,

    #[serde(default)]
    pub l1_checkpoint_proof: Option<L1Proof>,
}

// Custom default functions for non-trivial defaults
fn default_filter_hash() -> Bytes32 {
    EMPTY_ROOT
}
```

## Acceptance Criteria

- [ ] All fields added after protocol version 1 have `#[serde(default)]` or `#[serde(default = "...")]`
- [ ] Deserializing v1 data into the current struct succeeds
- [ ] Default values are sensible (e.g., `None` for Options, `EMPTY_ROOT` for hash fields)
- [ ] No deserialization failures when older data lacks new fields
- [ ] Custom default functions are provided where `Default::default()` is not appropriate

## Implementation Notes

- `#[serde(default)]` uses the type's `Default` implementation. For `Option<T>`, this is `None`. For numeric types, this is `0`. For `Bytes32`, this is all zeros.
- `#[serde(default = "path::to::function")]` allows custom defaults. Use this when the zero value is not the correct semantic default (e.g., `filter_hash` should default to `EMPTY_ROOT`, not `ZERO_HASH`).
- This is especially important for bincode, which does not have field names in the wire format. Field ordering must remain stable across versions.
