# STR-004: CoinLookup and BlockSigner Trait Definitions

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 7.2](../../../../resources/SPEC.md)

## Summary
The dig-block crate defines two key traits that abstract external dependencies: `CoinLookup` for querying on-chain coin state, and `BlockSigner` for producing BLS signatures over block hashes. These traits allow the crate to remain agnostic to the specific full-node RPC or key management backend used by consumers.

## Specification
### CoinLookup Trait

The `CoinLookup` trait MUST be defined in `src/traits.rs` with the following methods:

```rust
use chia_protocol::{Bytes32, CoinState};

pub trait CoinLookup {
    /// Look up the current state of a coin by its coin ID.
    /// Returns None if the coin is unknown.
    fn get_coin_state(&self, coin_id: &Bytes32) -> Option<CoinState>;

    /// Returns the current chain height as observed by this lookup source.
    fn get_chain_height(&self) -> u64;

    /// Returns the current chain timestamp as observed by this lookup source.
    fn get_chain_timestamp(&self) -> u64;
}
```

Key design decisions:
- The trait uses `chia_protocol::CoinState` directly rather than defining a custom wrapper, maximizing compatibility with existing Chia tooling.
- `get_coin_state` returns `Option<CoinState>` rather than `Result` because a missing coin is a normal condition (not an error) during validation.
- `get_chain_height` and `get_chain_timestamp` provide context needed for time-lock and height-lock condition validation during block execution.

### BlockSigner Trait

The `BlockSigner` trait MUST be defined in `src/traits.rs` with the following method:

```rust
use chia_bls::Signature;
use chia_protocol::Bytes32;

pub trait BlockSigner {
    /// Sign the given block hash, producing a BLS signature.
    /// Returns an error if signing fails (e.g., key unavailable).
    fn sign_block(&self, block_hash: &Bytes32) -> Result<Signature, SignerError>;
}
```

`SignerError` SHOULD be defined in `src/error.rs` or alongside the trait, and MUST capture at least: key-not-available and signing-failed error conditions.

Both traits MUST be object-safe to allow usage as `dyn CoinLookup` and `dyn BlockSigner` trait objects where needed.

## Acceptance Criteria
- [ ] `CoinLookup` trait is defined in `src/traits.rs`
- [ ] `CoinLookup::get_coin_state` accepts `&Bytes32` and returns `Option<CoinState>`
- [ ] `CoinLookup::get_chain_height` returns `u64`
- [ ] `CoinLookup::get_chain_timestamp` returns `u64`
- [ ] `CoinState` is imported from `chia_protocol`, not re-defined
- [ ] `BlockSigner` trait is defined in `src/traits.rs`
- [ ] `BlockSigner::sign_block` accepts `&Bytes32` and returns `Result<Signature, SignerError>`
- [ ] Both traits are object-safe (can be used as `dyn CoinLookup` / `dyn BlockSigner`)

## Implementation Notes
- Primary file: `src/traits.rs`
- Dependencies: `chia-protocol` (for `Bytes32`, `CoinState`), `chia-bls` (for `Signature`)
- Edge cases: Object safety requires that no methods return `Self` or use `Self` in type parameter position. The current signatures are object-safe by design. Future additions to these traits must preserve object safety.
