# STR-002: Module Hierarchy

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary
The dig-block crate MUST organize its source modules according to the hierarchy defined in SPEC Section 11. This structure separates concerns into types, validation, building, and shared utilities, ensuring the codebase remains navigable and each module has a single clear responsibility.

## Specification
The `src/` directory MUST contain the following module hierarchy:

```
src/
  lib.rs
  constants.rs
  error.rs
  hash.rs
  traits.rs
  types/
    mod.rs
    header.rs
    block.rs
    attested.rs
    checkpoint.rs
    receipt.rs
    signer_bitmap.rs
    status.rs
  validation/
    mod.rs
    structural.rs
    execution.rs
    state.rs
  builder/
    mod.rs
    block_builder.rs
    checkpoint_builder.rs
```

Module responsibilities:
- **types/**: All data structures representing DIG L2 block components
  - `header.rs` - `L2BlockHeader` struct and serialization
  - `block.rs` - `L2Block` (header + SpendBundle pair)
  - `attested.rs` - `AttestedBlock` (block + signer attestations)
  - `checkpoint.rs` - `Checkpoint` and `CheckpointSubmission`
  - `receipt.rs` - `Receipt`, `ReceiptList`, `ReceiptStatus`
  - `signer_bitmap.rs` - `SignerBitmap` for compact signer tracking
  - `status.rs` - `BlockStatus`, `CheckpointStatus` enumerations
- **constants.rs** - Protocol constants (version, magic bytes, limits)
- **error.rs** - All error enumerations (`BlockError`, `CheckpointError`, `BuilderError`, `SignerBitmapError`, `ReceiptError`)
- **hash.rs** - Block hashing utilities (header hash, block hash computations)
- **validation/** - Block and checkpoint validation logic
  - `structural.rs` - Field-level structural validation
  - `execution.rs` - CLVM execution and spend validation
  - `state.rs` - State transition validation
- **builder/** - Fluent builder APIs
  - `block_builder.rs` - `BlockBuilder` for constructing L2Block instances
  - `checkpoint_builder.rs` - `CheckpointBuilder` for constructing Checkpoint instances
- **traits.rs** - External integration traits (`CoinLookup`, `BlockSigner`)

Each subdirectory MUST contain a `mod.rs` that declares and re-exports its child modules.

## Acceptance Criteria
- [ ] All listed files exist in `src/`
- [ ] `types/mod.rs` declares modules: header, block, attested, checkpoint, receipt, signer_bitmap, status
- [ ] `validation/mod.rs` declares modules: structural, execution, state
- [ ] `builder/mod.rs` declares modules: block_builder, checkpoint_builder
- [ ] `lib.rs` declares all top-level modules: types, constants, error, hash, validation, builder, traits
- [ ] `cargo check` succeeds with the full module hierarchy

## Implementation Notes
- Primary file: `src/lib.rs` (top-level module declarations)
- Dependencies: Each `mod.rs` must properly declare and re-export submodules
- Edge cases: Module files may initially contain only type stubs or `todo!()` implementations, but the file and module declaration structure must be complete
