## Summary

Defines the `ReceiptStatus` enum representing the outcome of a transaction execution. Each variant has a fixed numeric representation for deterministic serialization.

## Specification

### Enum Definition

```rust
#[repr(u8)]
pub enum ReceiptStatus {
    Success = 0,
    InsufficientBalance = 1,
    InvalidNonce = 2,
    InvalidSignature = 3,
    AccountNotFound = 4,
    Failed = 255,
}
```

### Variant Descriptions

| Variant              | Value | Description                                           |
|----------------------|-------|-------------------------------------------------------|
| `Success`            | 0     | Transaction executed successfully.                    |
| `InsufficientBalance`| 1     | Sender did not have enough balance to cover the tx.   |
| `InvalidNonce`       | 2     | Transaction nonce did not match the expected nonce.   |
| `InvalidSignature`   | 3     | Transaction signature verification failed.            |
| `AccountNotFound`    | 4     | Sender account does not exist in the state.           |
| `Failed`             | 255   | Generic failure for other execution errors.           |

## Acceptance Criteria

- [ ] `ReceiptStatus` defines exactly six variants with correct names.
- [ ] `#[repr(u8)]` is applied for numeric representation.
- [ ] `Success` has value 0.
- [ ] `InsufficientBalance` has value 1.
- [ ] `InvalidNonce` has value 2.
- [ ] `InvalidSignature` has value 3.
- [ ] `AccountNotFound` has value 4.
- [ ] `Failed` has value 255.
- [ ] Numeric values survive round-trip conversion (u8 -> enum -> u8).

## Implementation Notes

- The `#[repr(u8)]` attribute ensures deterministic serialization in receipts and Merkle root computation.
- `Failed = 255` is intentionally separated from the sequential values to reserve space for future specific failure codes.
- Conversion from `u8` should handle unknown values gracefully (e.g., map to `Failed`).
